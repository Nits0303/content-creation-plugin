# Skill: Refine
**File:** skills/refine/SKILL.md
**Role:** Modify existing generated content based on user instruction. Does not restart intake. Does not rebuild the CONTENT_BRIEF.

---

## Purpose

Refine operates on content that has already been generated by the plugin pipeline. It applies targeted modifications based on user instruction — length, tone, or structure — without discarding or rebuilding the brief that produced the original content.

The existing CONTENT_BRIEF remains the system's source of truth throughout refinement.

---

## Pipeline Position

```
/create → intake → router → generation skill → humanise → [content delivered]
                                                                    ↓
                                                              /refine ← you are here
```

Refine is not part of the generation pipeline. It is triggered after content has been delivered.

---

## Command Trigger

Activated by: `/refine`

---

## Inputs

Refine receives three inputs:

1. **Existing CONTENT_BRIEF** — unchanged from the generation that produced the original content
2. **Previously generated content** — the full output delivered to the user
3. **User's refine instruction** — the modification request in plain language

Examples of valid instructions:
- "make this shorter"
- "make the tone more conversational"
- "add a section about pricing"
- "expand the explanation of step 3"
- "remove the intro"
- "condense the middle section"

---

## Iteration Behaviour Rules

These rules are locked and apply without exception.

- Refine does NOT restart intake.
- Refine does NOT ask intake questions.
- Refine does NOT rebuild the CONTENT_BRIEF.
- Refine does NOT change format, sub_type, or sector.
- Refine does NOT modify brand identity fields.
- Refine does NOT trigger sector inference.
- Refine does NOT modify SEO keyword logic.
- The existing CONTENT_BRIEF is authoritative. Only three fields may be adjusted. All other fields remain as set by intake.

---

## Instruction Classification

**Step 1** — Classify the user's instruction into one of three signal types before acting.

### Length Signal

Trigger phrases: "shorter", "longer", "expand", "condense", "cut this down", "make it longer", "trim"

Allowed CONTENT_BRIEF change: `length_tier`

### Tone Signal

Trigger phrases: "more casual", "more formal", "friendlier", "more authoritative", "less corporate", "warmer", "more conversational", "more professional"

Allowed CONTENT_BRIEF change: `tone`

### Structural Signal

Trigger phrases: "add a section about X", "remove the intro", "expand section on Y", "cut the conclusion", "restructure", "move X before Y", "add more examples"

Allowed CONTENT_BRIEF change: `content_points`

If a structural change significantly increases scope: `length_tier` may also adjust.

### Multiple Signals

If the instruction contains more than one signal type, apply changes in this order:

```
structure → length → tone
```

---

## Allowed CONTENT_BRIEF Adjustments

Refine may modify only these three fields:

| Field | When modified |
|---|---|
| `length_tier` | Length signal detected or structural scope expands significantly |
| `tone` | Tone signal detected |
| `content_points` | Structural signal detected — section added, removed, or expanded |

Refine may modify only `length_tier`, `tone`, and `content_points`. All other CONTENT_BRIEF fields are read-only during refinement.

Refine must not modify the `keyword` field. If the user requests a keyword change, the request must be delegated to the SEO skill rather than handled within refine.

---

## Structural Consistency Rules

Refine preserves the structural requirements defined by the generation skill for the active subtype. Subtype architecture belongs to the generation skills (`skills/blog/SKILL.md`, `skills/webpage/SKILL.md`, `skills/social/SKILL.md`). Refine does not redefine subtype architecture.

Core components for the active subtype must remain present after any refinement operation. Enrichment components may be added or removed. Supporting components may be added or removed within the selection cap defined by the generation skill.

Refine must never reduce content below the minimum required to populate all Core components for the subtype.

---

## Hallucination Safeguards

F.0.3 rules apply to all refine operations without exception. Refinement does not relax hallucination standards.

Refine must not introduce:
- Fabricated statistics or data
- Fabricated or attributed quotes
- Synthetic research claims ("studies show", "experts agree", "recent data suggests")
- Named sources that cannot be traced to a verified source already present in the content

**Refine does not proactively run research.** External research belongs to generation skills during the original generation step. Refine reuses existing verified material already present in the content.

If the user explicitly requests new evidence, statistics, or external examples, refine may invoke the same sourcing behaviour used by the generation skill, following the F.0.3 Source Verification Chain. Refine does not introduce a separate research system — it uses the same verification rules defined for generation.

When sourcing is invoked under F.0.3:
1. Trace the claim to a specific URL or named publication retrieved in the current session.
2. If a reliable source is found: include the material with source attribution.
3. If web search returns no credible source: drop the section entirely or insert a placeholder — `[Include your own source here — web search returned no credible results for this component]`.

Never approximate. Never use training data recall as a citable source.

---

## Controlled Regeneration

Refine regenerates only the sections directly affected by the instruction. All unchanged sections are passed through intact.

### Length modification — compress

- Remove Enrichment components first.
- Remove Supporting components next if compression target is not met.
- Shorten individual sections by reducing examples to their most essential version.
- Core components are never removed regardless of compression target.
- If the user requests compression that would require removing Core components: flag the constraint before proceeding. Do not silently drop Core content.

### Length modification — expand

- Deepen explanation in existing sections before adding new sections.
- Add Supporting or Enrichment components where material exists or can be sourced.
- Do not pad. Do not add generic filler. Length increase must come from substantive additions.
- Refine must not increase `length_tier` beyond what the existing content and user-provided instruction can realistically support. If the requested expansion would require additional information not present in the brief or the content, flag the missing material before proceeding. Do not fabricate content to fill a higher tier.

### Tone modification

- Adjust vocabulary, sentence register, and formality level throughout.
- Do not alter meaning, claims, statistics, quotes, or CTAs.
- Apply the updated `tone` value through the brand-voice calibration rules.
- Tone change applies to all sections, not only the sections the user mentions.

### Structural modification — add section

- Determine correct placement within the subtype's section hierarchy.
- Check whether the addition requires sourced material — apply hallucination safeguards.
- Adjust `content_points` in the CONTENT_BRIEF to reflect the addition.
- If addition expands scope significantly: update `length_tier` accordingly.

### Structural modification — remove section

- Remove the specified section.
- Check whether remaining content still satisfies all Core component requirements for the subtype.
- If removal creates a Core component gap: flag before proceeding.

### Structural modification — expand section

- Deepen the specified section only.
- All other sections remain unchanged.
- Apply hallucination safeguards to any new material added.

---

## Humanise Interaction

After controlled regeneration is complete, the refined output passes through `skills/humanise/SKILL.md`.

The same HUMANISE_PASS value assigned during original generation applies to the refined output. Refine must preserve the HUMANISE_PASS value attached to the original content. Refine must not attempt to determine or recalculate HUMANISE_PASS. The humanise skill will apply the same pass mode during the refinement output stage.

Humanise rules remain unchanged:
- Linguistic refinement only
- No structural changes
- No factual changes
- No CTA modification
- No citation modification
- All preservation rules apply

---

## Output Rules

Refine returns the fully updated version of the content.

- Output the complete refined content — not a diff, not a summary of changes.
- Do not explain what was changed.
- Do not add commentary.
- Do not add a preamble.
- Return the content ready for delivery to the user.

---

## Prohibited Behaviours

This skill must never:

1. Trigger intake or ask intake questions
2. Discard or rebuild the existing CONTENT_BRIEF
3. Modify brand identity fields (`brand_name`, `brand_description`)
4. Change `format`, `sub_type`, or `sector`
5. Trigger sector inference
6. Modify SEO keyword logic
7. Bypass F.0.3 hallucination safeguards
8. Rewrite the entire document when only one section was requested
9. Remove Core components to satisfy a length compression request without flagging
10. Introduce fabricated statistics, quotes, or synthetic authority claims
11. Reassign or change HUMANISE_PASS
12. Output a diff or change log instead of the complete refined content
