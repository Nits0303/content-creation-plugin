# /refine

---

## Command Purpose

`/refine` is the entry point for modifying previously generated content. It signals that the user wants to adjust an existing output without restarting the pipeline. The command triggers the refine skill. It contains no refinement logic.

---

## When This Command Is Used

Use `/refine` after content has been generated by `/create`, `/create-blog`, `/create-page`, or `/create-social`. It is used when the user wants to modify the existing output rather than generate new content from scratch.

Typical triggers include:

- Adjusting length — "make it shorter", "expand this"
- Adjusting tone — "more formal", "more conversational"
- Adjusting structure — "add a section about X", "remove the introduction", "reorder the points"

`/refine` is not used to change the content format. If the user wants to convert existing content into a different format, use `/repurpose`.

---

## Refinement Behaviour

`/refine` does not restart intake. It does not create a new `CONTENT_BRIEF`. It operates on the `CONTENT_BRIEF` constructed during the original generation.

The user's refinement request is interpreted as a signal. Each signal type maps to a specific field modification inside the existing brief:

- **Length signal** — modifies `CONTENT_BRIEF.length_tier`
- **Tone signal** — modifies `CONTENT_BRIEF.tone`
- **Structural signal** — modifies `CONTENT_BRIEF.content_points`; may also adjust `CONTENT_BRIEF.length_tier` if the scope of the content changes

All other `CONTENT_BRIEF` fields remain unchanged. The original format is preserved.

---

## Pipeline Activation

Invoking `/refine` starts the following pipeline:

```
/refine → refine/SKILL.md → [original generation skill] → humanise/SKILL.md
```

The generation skill activated is determined by `CONTENT_BRIEF.format` from the original generation:

```
format = blog    → blog/SKILL.md
format = webpage → webpage/SKILL.md
format = social  → social/SKILL.md
```

The router is not re-entered. Format and subtype are read from the existing `CONTENT_BRIEF`.

---

## What Happens After The Command Runs

1. `refine/SKILL.md` opens and loads the existing `CONTENT_BRIEF`.
2. The user's refinement request is interpreted as one or more signals (length, tone, or structural).
3. The relevant `CONTENT_BRIEF` fields are updated to reflect the signals. All other fields remain unchanged.
4. The original generation skill (`blog/SKILL.md`, `webpage/SKILL.md`, or `social/SKILL.md`) regenerates the content using the updated brief.
5. `humanise/SKILL.md` runs the humanise pass on the regenerated output.
6. The refined content is returned to the user.

---

## Output Behaviour

- Output format and subtype match the original generation. Neither changes during refinement.
- Output reflects only the fields modified by the refinement signal. Unmodified fields produce the same structural and tonal behaviour as the original.
- If a structural signal adds or removes content points, output length may change even if no explicit length signal was given.
- Sector overlay and brand voice behaviour carry over from the original `CONTENT_BRIEF` without re-evaluation.

---

## Restrictions

- This command file contains no refinement logic, brief modification logic, generation logic, template logic, SEO logic, humanise logic, or sector overlay logic.
- All logic lives in the skill files activated by this pipeline.
- Do not modify this command file to add refinement behaviour.
- Do not use this command to change content format. Use `/repurpose` for format conversion.
- Do not use this command to generate new content from scratch. Use `/create`, `/create-blog`, `/create-page`, or `/create-social` for new generation.
